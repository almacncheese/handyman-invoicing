HandyQuote — Code & Architecture Audit

A ground-up review of the AI-built (Grok) multi-tenant SaaS at quickhandyquote.com — security, data integrity, scalability, and product UX — read against the actual running application and source, not the builder's own notes.
Repo almacncheese/handyman-invoicing Stack Next.js 16 · Prisma/Postgres · TypeScript Status Live, real trial signups Reviewed July 2026

Executive Summary

HandyQuote is a genuinely competent build — better, in several dimensions, than the average human-written MVP. Multi-tenant authorization is applied with unusual consistency across all 30+ API routes; money is modeled correctly as integer cents end-to-end; the public-facing e-signature link uses strong cryptographic tokens with no XSS surface anywhere; and the server/client component split shows real architectural judgment rather than the common "everything is a client component" anti-pattern. Al's engineering lessons clearly made it into this build: fail-closed secrets, tenant isolation via 404-not-403, atomic payment claims — the patterns are there, and mostly they hold up when we went and checked the actual code rather than trusting the builder's own notes about itself.

That said, this is not a codebase to hand paying customers' data to without changes. We found four critical-severity issues, the worst of which is a live, unscoped database-wipe script one environment-variable misconfiguration away from deleting every tenant's data — paired with the fact that no backup strategy exists at all. Two more critical bugs are genuine data-integrity races in the e-signature/status lifecycle: a signed, accepted estimate can be silently flipped back to "declined," and a voided estimate can be resurrected into a real invoice. For a product whose entire value proposition is "a legally meaningful customer signature," these are not cosmetic.

Nineteen high-severity findings round out the picture: rate limiting across every auth and public endpoint is trivially bypassed via a spoofed header; the manual-payment ledger can desync from the real invoice balance under concurrency; there's zero test coverage on any of the 34 API routes and zero CI; and the entire deployment is one VPS, one app container, one database container, with no replicas and no monitoring. None of this is unfixable — several of the fixes are single-file changes — but none of it should be discovered by a customer first.

Bottom line: strong bones, real cracks. The security and multi-tenancy foundation is above average for an AI-built SaaS. The data-loss and data-integrity risks are not — and given the product is already live with real signups, we'd treat the four critical items as a stop-ship punch list, not a backlog.

Critical Findings

All four independently re-derived by us from the actual route code — not taken on the strength of the automated review alone. If we fix nothing else, fix these before onboarding another paying tenant.
Critical · Data Loss

One unscoped seed script can wipe every tenant's data in production, and there is no backup to recover from

prisma/seed.ts gates a full-database wipe (business.deleteMany() cascading through users, customers, quotes, invoices, payments, activity) behind a single condition: SEED_WIPE === '1' || NODE_ENV !== 'production'. Today's Docker build correctly pins NODE_ENV=production, so the app is safe as currently deployed — but AGENT_NOTES.md's own history log already records one prior incident of exactly this kind of environment-config drift ("Init migration lagged product schema"). There is no soft-delete anywhere in the schema, no confirmation prompt, and no scoping by business id. And separately — we found no backup mechanism of any kind, anywhere — no pg_dump cron, no managed-Postgres snapshot, no offsite export. A single Docker volume on a single VPS is the only copy of every customer's estimates, signatures, and payment records that exists.

prisma/seed.ts:21 — const wipe = SEED_WIPE === '1' || NODE_ENV !== 'production'; ...line 31 — await prisma.business.deleteMany(); // cascades everything docker-compose.yml — postgres_data is a local named volume; no backup service defined anywhere in the repo

Fix: scope the wipe to a specific demo business id/slug (never a global deleteMany), require an explicit CLI flag rather than an env-var default, add a deletedAt soft-delete column, and stand up an actual pg_dump schedule with offsite retention before treating this as safe for more paying tenants.

Critical · Data Integrity

A customer's signed acceptance can be silently overwritten back to "declined"

The accept endpoint correctly guards its write with a conditional update so two near-simultaneous requests can't clobber a signature. The decline endpoint does not: it reads status once, checks it, then writes status: 'declined' unconditionally with no re-check at write time. If an accept commits between decline's read and write — a double-click, a slow network retry — decline still fires, leaving a row that is simultaneously "declined" in status while carrying a fully signed acceptedAt/signatureData. For a product whose core value is a legally meaningful e-signature, a self-contradictory accepted-but-declined record is a real liability, not a UI glitch.

src/app/api/public/estimate/[token]/decline/route.ts — plain prisma.quote.update(), no where-clause guard on current state vs. .../accept/route.ts — correctly uses updateMany({ where: { id, acceptedAt: null, signatureData: null } })

Fix: mirror the accept route's pattern exactly — a conditional updateMany guarded on acceptedAt: null, checking result.count === 0 to detect a lost race.

Critical · Data Integrity

A voided estimate can be resurrected into a real invoice

The status model treats "void" as terminal — nothing should transition out of it. Voiding a quote, however, never clears its acceptedAt timestamp. The invoice-conversion route ignores the shared state machine entirely and instead checks if (quote.acceptedAt && status !== 'accepted' && status !== 'invoiced') — which is true for a quote that was accepted, then later voided. Calling convert on that quote id silently flips it back to "accepted" and proceeds to generate an invoice, completely bypassing the void terminal-state guarantee the rest of the codebase relies on.

src/app/api/quotes/[id]/convert/route.ts — acceptedAt-based bypass, no canTransition() call, no check for status === 'void' src/app/api/quotes/[id]/void/route.ts — voiding never clears acceptedAt src/lib/quote-status.ts — canTransition() explicitly blocks every transition out of 'void'

Fix: have convert call the shared assertTransition() instead of its own acceptedAt heuristic, and/or clear acceptedAt/signatureData when a quote is voided.

Critical · Data Loss

The database-wipe script has no scoping, no confirmation, and no per-tenant awareness

Separately from the backup gap above, the wipe logic itself is worth flagging on its own terms: it is a single boolean env-var check standing between "reseed the demo account" and "delete every business that has ever signed up." There is no allowlist of environments, no dry-run, no confirmation, and no way to scope it to one tenant. This is exactly the class of environment-config mistake AGENT_NOTES.md's own history log already records happening once in this project (a migration lagging production schema) — the fix belongs in the script itself, independent of whatever backup story gets built.

prisma/seed.ts:21-31 — single env-var branch gates an unscoped deleteMany() chain across Activity, Payment, Invoice, Quote, LineTemplate, Customer, User, Business

Fix: require an explicit --i-mean-it-style CLI flag (never an env-var default) to run the wipe branch at all, and scope every delete to a specific business id/slug rather than a bare deleteMany().

High-Severity Findings

Nineteen findings, grouped by area. Five of the six security-tagged items below were independently adversarially re-verified by a second pass of reviewers instructed to try to disprove them — all five survived.
Security
High ✓ verified

Rate limiting on every auth endpoint is bypassed by spoofing one header

Login, signup, forgot-password, reset-password, and change-password are all rate-limited — but the limiter's identity key is the client-supplied X-Forwarded-For header, taken verbatim with no trusted-proxy validation. Any caller can send a different fake IP on every request and reset their own bucket every time, fully defeating the limit. There's no secondary per-account backstop either.

src/lib/rate-limit.ts — clientIp() returns the first comma-separated X-Forwarded-For value with no hop-count/allowlist check Used identically in all 5 auth routes: rateLimit({ key: 'login:'+ip, ... })

Fix: only trust X-Forwarded-For behind a known proxy, pick the correct trusted hop, and add a per-account limiter as a backstop independent of IP.

High ✓ verified

The public estimate link has the same spoofable limiter — and two of its three routes have no rate limit at all

Only the accept endpoint calls the rate limiter; the GET lookup and the decline endpoint have none. The one limit that does exist is bypassable the same way as the auth endpoints. Given the token's 192-bit entropy this isn't a practical brute-force vector today, but it does mean unlimited unauthenticated reads and unlimited decline-spam against any token an attacker obtains through another channel.

src/app/api/public/estimate/[token]/route.ts — no rate-limit import .../decline/route.ts — no rate-limit import .../accept/route.ts — the only one that calls rateLimit(), and it's spoofable per the finding above

Fix: add the same rate-limit call used on accept to decline and the GET lookup, once the underlying IP-spoofing issue is fixed.

High

User.email has no global-unique constraint — a signup race can register two accounts under one email

The only DB constraint is composite (businessId, email); email alone is just indexed. Signup does a check-then-create against a brand-new business id each time, so two concurrent signups with the same email both pass the pre-check and both succeed as separate tenants. Login then uses findFirst (not findUnique) on email alone, so it silently resolves to whichever row Postgres happens to return — the other account's owner may never reach their own workspace via that email again. The identical pattern exists in team invite and admin user creation.

prisma/schema.prisma — @@unique([businessId, email]) only, no bare unique on email src/app/api/auth/signup/route.ts — findFirst check, then a separate transaction creating a new business + user with no re-check inside it

Fix: add a DB-level unique index on email alone, or re-check inside the signup transaction / catch the resulting constraint violation.

High ✓ verified

Manual payment recording can desync the invoice balance from the real payment ledger

The idempotency key that's supposed to make payment recording safe against duplicate submissions is optional, and its only caller (the "Record cash/Zelle payment" button) never sends one — so a genuine double-click creates two distinct payment rows, not one deduplicated one. Separately, the balance update reads the invoice once, computes a new absolute value in application code, and writes it back with no row lock — two concurrent recordings against the same invoice can both read the same stale balance and each overwrite the other's update, so the Payment table can show two successful payments while the invoice's cached balance reflects only one.

src/app/api/payments/record/route.ts — key falls back to a fresh random value when absent; balance write is a stale-read-then-absolute-write, not an atomic increment src/components/QuoteActions.tsx — recordOffline() never sends an idempotencyKey

Fix: generate the idempotency key client-side and persist it across retries; make the balance update an atomic conditional (increment, or SELECT ... FOR UPDATE) rather than a read-then-write.

Data Model & Business Logic
High

The documented status state machine is enforced in exactly one route out of six

quote-status.ts is described in the project's own notes as "the" forward-only lifecycle gate. In the actual code it's imported by value in exactly one route (void). Every other status-mutating endpoint — the PATCH lock, accept, decline, send, and invoice conversion — reimplements its own hand-written status whitelist. None of these are cross-checked against each other, which is exactly how the two critical race-condition findings above happened: the shared, well-tested module gives false confidence about enforcement it doesn't actually provide everywhere.

grep for imports of quote-status.ts across src/ → only void/route.ts (value import); every other route hand-writes its own status array

Fix: route every status write through the shared assertTransition() (or a wrapping transitionQuote() helper), deleting the duplicated inline arrays.

High

Quote and invoice numbering are non-atomic read-then-write — racy under concurrent creation

Both counters are read via a plain query, incremented in JavaScript, then written back as a literal value rather than a DB-level atomic increment. A unique constraint prevents a silently duplicated number, but the losing concurrent request just fails with an unhandled, confusing 400 rather than gracefully retrying — real risk any time two staff create estimates back-to-back or a request gets double-submitted.

src/app/api/quotes/route.ts & .../[id]/duplicate/route.ts — nextQuoteNumber + 1 computed in JS, written as a literal src/app/api/quotes/[id]/convert/route.ts — identical shape for invoice numbers

Fix: push the increment into the database itself (a raw RETURNING update, or Prisma's { increment: 1 }) so concurrent transactions serialize correctly.

High

Photo count/size limits exist in code but are never enforced on the write path

Unlike line items — which are fully re-validated on every write — an 8-photo cap and a per-image size guard are defined but never called from the actual quote create/update API. Both routes accept a photos array with only shape validation and no length or size limit, then write it straight to the JSON column. A direct API call (bypassing the browser UI, which does enforce the cap client-side) can attach an unbounded number of arbitrarily large embedded images to a single quote row.

src/lib/photos.ts — assertCanAddPhoto() defined, zero call sites anywhere in src/ src/app/api/quotes/route.ts & .../[id]/route.ts — photos Zod schema has no .max() on array length or string size

Fix: call the existing guard (or add matching Zod .max() constraints) inside the POST/PATCH handlers before persisting photos.

Testing, CI & Observability
High

Zero route-level tests across all 34 API endpoints

Every one of the 34 route.ts handlers — auth, payments, quotes, invoices, admin, the public e-signature endpoints — has no integration test at all. Only some of the pure library functions they call are tested in isolation; the actual request parsing, status mapping, and wiring between session/authz/db at the route boundary is entirely unverified.

find src/app/api -name route.ts → 34 files; matching *.test.ts → 0

Fix: start with the highest-risk handlers — login, payment recording, the three public accept/decline/pay routes, and invoice conversion.

High

No CI pipeline — tests and build checks are a manual, unenforced local step

There is no .github/workflows directory or any other CI configuration. The only gate against shipping a broken build or a bad migration is a manual checklist bullet in the deploy notes ("npm test + npm run build"). Nothing stops a change that breaks the payment or authz logic from being pushed straight to the single production container.

No .github/ directory anywhere in the repo; DEPLOY.md's "Redeploy" instructions are simply "push main (auto-deploy on)"

Fix: add a GitHub Actions workflow running test/typecheck/build on every push, gating the Coolify auto-deploy.

High

No end-to-end test layer of any kind

No Cypress, no Playwright, no e2e directory. The entire customer-facing surface — the quote builder, and critically the public accept/decline/pay flow real customers use to sign — has zero automated browser-level verification.

package.json devDependencies — vitest only; no e2e/cypress/playwright directories anywhere

Fix: a small Playwright smoke suite covering login and the public accept/decline flow would cover the highest-value, least-tested surface first.

High

No error-tracking SDK and almost no server-side error logging anywhere

No Sentry/Datadog/OpenTelemetry or equivalent exists in the dependency tree. Of 47 files with a try/catch, 29 of the 30 API routes funnel every unexpected exception into a shared helper that converts it straight to a JSON response with zero console output. A full grep of the source tree found exactly two console.error calls total. A genuine production failure — a Prisma error, a DB blip — leaves no trace anywhere a human would see it.

grep console.error src/ → 2 matches, both in email.ts; grep for Sentry/Datadog/etc in package.json → none

Fix: add a minimal error-tracking SDK, and log (not just relay) any exception that doesn't carry an explicit, intentional status code.

High

No error boundary anywhere in the app — a component crash mid-estimate has no recovery screen

Next.js's App Router supports a custom error.tsx/global-error.tsx at any route level. Neither exists anywhere in this app — only a 404 page. An uncaught render exception on any page (the quote builder mid-estimate, an invoice view) falls through to Next's bare default error handling, with no branded recovery screen and no "try again" affordance.

find src/app -iname '*error*' → only not-found.tsx

Fix: add a root global-error.tsx and, for the quote builder specifically, a route-level error.tsx with a save-and-retry affordance.

Scalability & Operations
High

No monitoring or alerting — the health check exists but nothing automated polls it

The health endpoint is a real, DB-backed check (a genuinely good implementation), but nothing in the repo calls it on a schedule. The only caller is a manual smoke-test script a human runs by hand after a deploy. An outage, a crash-looping container, or a stuck migration would surface via a customer complaint before it surfaces any other way.

src/app/api/health/route.ts — legitimate SELECT 1 check; no cron/CI/uptime-monitor caller anywhere in the repo

Fix: point an external uptime monitor (even a free one) at /api/health with alerting to a phone/Slack.

High

Migrations run at container boot with no rollback plan — a bad migration takes the only production instance fully down

The startup script runs prisma migrate deploy before starting the server, under set -eu. If a migration fails for any reason, the container never reaches "serving," and — since there's only one container — the app goes down with no automatic fallback and no documented rollback procedure. This exact failure mode (migration/schema drift) already happened once in this project per its own history log.

scripts/docker-start.sh — set -eu; prisma migrate deploy; exec npm start No rollback/blue-green/canary procedure documented anywhere

Fix: at minimum, decouple migration from container boot (run it as a separate deploy step you can watch), and document a rollback procedure before the next schema change.

High

Single VPS, single app container, single database container — no replicas, no resource limits, no failover

The entire system runs on one box: one Coolify app, one Postgres container, one nginx edge, with no CPU/memory limits and no clustering. A crash, an OOM, or a reboot of that one box takes down the app and the database simultaneously. There's no load-test data in the repo, but the architecture points to Postgres round-trips as the first real ceiling, well before raw CPU.

DEPLOY.md "Live topology" — one app, one DB, one nginx vhost, all on one VPS, no replica fields

Fix: see the scalability roadmap below — this isn't urgent at current traffic, but it's the wall the product will hit first as it grows.

High

List endpoints and the dashboard have no pagination — data is silently truncated, not just slow

No endpoint anywhere reads a skip/cursor/page parameter. Several routes hardcode a result cap with no way to reach older rows and no signal to the UI that anything is hidden. Two dashboard queries have no limit at all and pull every quote and invoice a tenant has ever created into Node just to sum a few numbers on every single page load — the endpoint every user sees first after login is also the one most likely to slow down as tenants accumulate history.

src/app/dashboard/page.tsx — two unbounded findMany() calls run on every dashboard load GET /api/quotes, /api/invoices — hardcoded take:200 with no total-count signal or next-page mechanism

Fix: add real (cursor-based) pagination to the list endpoints, and replace the dashboard's unbounded fetch-then-reduce with a Postgres aggregate/groupBy so the database does the summing.

What They Got Right

A due-diligence review that only lists problems isn't honest. Fifty distinct strengths were identified and independently checked against the code — here are the ones worth a client actually knowing about, because they're the reason the critical findings above are fixable bugs in a good foundation, not a foundation to rebuild.

Security & Multi-Tenancy
  • Tenant isolation is applied with real consistency across all 16 core business routes — fetch-by-id, then assertSameBusiness(), then act, every time, including in less-obvious spots like nested activity queries and invoice-sequence generation where it would have been easy to skip.
  • Cross-tenant lookups correctly 404 instead of 403, deliberately avoiding an existence-leak — and this reasoning is applied uniformly, not just claimed in the docs.
  • Sessions re-verify against the live database on every request rather than trusting the JWT's embedded claims, closing the common "demoted user's old token still has elevated access" bug class.
  • The production auth secret genuinely fails closed — the app throws at boot if AUTH_SECRET is missing in production, with the insecure dev fallback reachable only outside production.
  • Password reset tokens are textbook-correct: 256-bit random, hashed at rest, one-hour expiry, single-use, invalidated by a direct password change.
  • Signup cannot self-assign platform-admin, role, or a foreign business id — traced every write path that could set the admin flag; only an existing admin can grant it.
  • Zero Math.random() anywhere near a security-sensitive value — every real token uses crypto.randomBytes.
  • The public e-signature token is 192 bits of real entropy, shape-guarded before any database query, with an opaque 404 either way.
  • The public estimate response is a genuinely minimized field set — no internal notes, no other customers' data, no raw signature image, no business financial internals.
  • Zero dangerouslySetInnerHTML anywhere in the codebase — every user-controlled string goes through React's default escaping; payment-app links are scheme-locked and URL-encoded.
  • Card payments are genuinely, unconditionally disabled — the charge route is a 13-line stub that never touches the request, matching the product's own stated scope exactly.
  • The middleware correctly limits itself to a UX redirect, explicitly documented as not being the authorization boundary — exactly the right mitigation for a well-known Next.js middleware-only-auth trap.
  • The Docker image runs as a non-root user with a health check that actually queries Postgres rather than faking a 200.
  • No secret values anywhere in the repo or git history — a full pattern sweep came back clean, and the team's own secrets-handling scripts consistently avoid printing values.
Money, Data Model & API Design
  • Every monetary value is an integer cent, end to end, with no float arithmetic anywhere in the calculation chain — a detail many production apps still get wrong.
  • Totals are always recomputed server-side; the create/update schemas don't even accept a client-submitted total field, so there's no path for a spoofed price.
  • Payment idempotency is a real database unique constraint, not just an application-level check — a genuine key collision cannot double-credit, full stop.
  • Cascade design on Quote/Invoice shows real referential-integrity thinking: an invoiced quote can't be deleted at the database level, and the app independently blocks deleting a customer who has quote history — belt and suspenders, not an accident.
  • Zod validation is applied to literally every route that accepts a body — 19 out of 19, verified by cross-check, directly contradicting the usual "some routes trust raw JSON" concern.
  • Zero any types, zero @ts-ignore, zero TODO/FIXME markers anywhere in the 34-route API layer under strict TypeScript — built to satisfy the compiler, not around it.
  • Money-critical multi-table writes are correctly wrapped in real database transactions with in-transaction re-reads to guard against races — contrary to what you'd expect from a naive AI-generated app.
  • No N+1 query patterns anywhere — relation data is consistently loaded via Prisma's include/select rather than looped per-row queries.
Frontend & Product Craft
  • The server/client component split is genuinely deliberate — 18 of 20 pages are server components doing direct data fetches, with "use client" reserved for real interactive islands, avoiding the common "everything is a client component" AI-codebase smell.
  • The e-signature pad uses native Pointer Events, correctly unifying mouse, touch, and pen input in one code path — exactly right for a product whose core moment happens on a customer's phone.
  • Status badges never rely on color alone — the status word is always shown as visible text alongside the color dot.
  • Focus states are handled correctly, not stripped — every place a default outline is removed, an explicit focus-visible ring replaces it.
  • Email templates consistently escape every dynamic field, and every link is server-built from internal tokens, never from anything a user supplies.
  • The mobile app shell uses a real bottom tab bar, not a squeezed-down desktop layout — a deliberate, correct mobile-first pattern (see our live walkthrough below).

Medium-Severity Findings

Thirty-one findings at this level in total; the ones below are the ones we'd actually schedule work against. The rest are smaller, mostly-cosmetic variants of the same themes.

Medium

No session revocation on logout or password change

Sessions are stateless 14-day JWTs. Logging out only clears the cookie client-side — a previously issued token (stolen device, leaked browser cache) stays fully valid elsewhere for its full remaining life.

Medium

The real platform-admin account defaults to a hardcoded weak password

The script that provisions the actual production superadmin falls back to the literal string password1 with no production guard, unlike the (correctly fail-closed) auth secret — and a companion script's own comments suggest this fallback has actually been relied on operationally.

Medium

Photo uploads trust the client-declared file type for the stored Content-Type header

An image/svg+xml upload is accepted and stored with that exact Content-Type under a .jpg-suffixed key — if that object URL is ever opened directly rather than embedded, a browser would render it as SVG and execute any embedded script.

Medium

The audit trail never records IP address or user agent

Not even on the "customer accepted" entry that matters most for a signature dispute — despite the accept route already computing the client IP a few lines away for its rate limiter.

Medium

The activity log silently swallows every write failure

A systemic audit-log outage would be completely invisible — every quote action would keep appearing to succeed while the audit trail quietly goes blank, with no log line anywhere.

Medium

Two dashboard-critical list queries sort by a column with no matching index

Quotes and invoices both list by "most recently updated/created," but the schema's indexes don't cover that sort key — invisible today, a real cost as tenant history grows.

Medium

Migration history has already drifted from the schema file

Four columns created in the very first migration were never dropped and don't exist on the current model — harmless today only because the app never reads them, but a landmine for the next schema change.

Medium

Every authenticated request re-verifies identity against Postgres

A correct-for-safety design choice, but it means read traffic generates the same database load as writes — combined with the single, un-pooled Postgres container, this is the most likely first thing to saturate under real concurrent load.

Scalability & Path to Scale

The good news first: sessions are stateless JWTs (no shared session store needed to add app replicas), and the rate limiter is cleanly seamed behind a two-function interface used identically everywhere — swapping it for Redis is a single-file change, not a rewrite. The architecture doesn't hit one dramatic wall so much as it currently stacks every kind of load — reads, writes, and identity checks — onto one shared, unpooled Postgres container on one VPS with no second instance of anything.

Before more customers

Close the data-loss gap

Stand up a real pg_dump schedule with offsite retention, and fix the seed script's wipe guard. This is the one item on this entire report we'd treat as non-negotiable before onboarding another paying tenant.

Next

Add real pagination and push aggregation into SQL

The dashboard and reports pages currently pull a tenant's entire history into Node on every load. This is invisible at today's scale and will be the first thing a growing tenant notices.

Before a second replica

Move rate limiting off in-process memory and set a connection pool size

Both are currently correct for exactly one app instance. Adding a second replica without doing this first would silently double-count nothing on rate limits and risk exhausting Postgres's connection limit.

When it matters

Decouple migrations from container boot; add a managed/pooled Postgres

The harder, genuinely architectural work — worth doing before the single-VPS topology becomes the thing standing between the product and its next tier of customers.

Low-Severity & Nitpicks

Thirty-two smaller items, condensed. None of these are urgent on their own.

  • Login's response timing is measurably slower for a registered email (bcrypt only runs on a match) — a minor account-enumeration signal, though signup's own error message already discloses the same thing more directly.
  • Uncaught exceptions return the raw internal error message to the caller, including on public, unauthenticated routes — an information-hygiene gap, not an authorization bypass.
  • The security-critical session, token, and password-reset modules have no unit tests of their own, unlike the tenant-isolation and rate-limit modules.
  • One doc claims a timing-safe token comparison that isn't actually implemented — low-risk in practice given the token's entropy, but a documentation/reality mismatch worth correcting.
  • A genuine idempotent-retry race on payment recording gets a raw error instead of the intended "already recorded" replay response.
  • A moderate advisory exists in a dependency Next.js bundles internally — not fixable from this project's own package.json.
  • No Content-Security-Policy or Strict-Transport-Security header is set at the application layer (may be handled at the reverse proxy, which isn't part of this repo).
  • Currency is effectively hardcoded to USD — reasonable for a US-only MVP, but a real migration if multi-currency is ever needed.
  • The reports endpoint loads full quote/invoice/payment history per request with no date bound — fine at today's per-tenant scale, a real cost later.
  • Several confirm/delete actions use the browser's native confirm()/prompt() dialogs, breaking the app's otherwise consistent design language.
  • A handful of photo-remove buttons share an identical, non-differentiating accessible name for screen-reader users.
  • The signature canvas has no keyboard-accessible path — a real gap for the one action in the product that's legally meaningful.

Methodology

This audit combined an automated multi-reviewer pass with direct manual verification, on the principle that an AI-authored codebase's own documentation about itself (AGENT_NOTES.md, DECISIONS.md, PRODUCT_CONTRACT.md) is a claim to check, not a fact to inherit — several findings above exist specifically because a documented claim didn't match the code we actually read.

Static review: fourteen independent specialist passes covered authentication/session security, multi-tenant authorization (split across two passes covering all 30 API routes), payment and money-handling correctness, the public unauthenticated surface, secrets/deployment/dependency posture, API-layer consistency, the business-logic/domain layer, frontend component architecture, schema design, query and scaling patterns, infrastructure/deployment maturity, test coverage, and observability. Every finding required a specific file and line as evidence, not a general impression.

Adversarial verification: every finding rated critical or high from the security-focused passes was independently re-reviewed by two additional reviewers instructed to actively try to disprove it by re-reading the cited code themselves. All five that reached this stage survived. The four critical findings from other passes (which weren't part of that specific pipeline) were instead personally re-derived and traced end-to-end through the actual route code by the lead reviewer before being included here.

Live verification: we additionally stood up the application locally — a real Postgres database, a seeded demo workspace, the actual dev server — and drove the complete customer journey ourselves in both desktop and mobile viewports: account login, estimate creation with live total calculation, sending, opening the public link, drawing a signature with the real pointer-event pipeline, accepting, converting to an invoice, and recording a manual payment. Two apparent visual bugs surfaced during this process and were discarded after confirming, via direct DOM inspection, that they were artifacts of our own screenshot tooling rather than the application.
